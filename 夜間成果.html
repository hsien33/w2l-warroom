機制① 派工器（Dispatcher）

核心原則（Goldratt TOC）：佇列就是緩衝（buffer），補位就是緩衝管理。任務不靠「記得喚起」存活，靠「無狀態 dispatcher 每輪從零重建真相」存活。喚起會漏；掃描 + 對賬不會漏。
核心原則（Ohno 自働化）：重試耗盡的任務 stop the line——標 dead + 告警，絕不靜默丟失。

1.1 任務檔 schema（ops/queue/<task_id>.json）

task_id 命名為決定性指紋——同一邏輯任務永遠同名 → 重覆派工只覆寫同檔，不生第二筆 = 第一層防雙發：

<yyyymmdd>-<lane>-<slug>     例：20260621-fbpost-day011 / 20260621-warcard-day010

{
  "task_id": "20260621-fbpost-day011",
  "lane": "fbpost",
  "assignee": "w2l-fb-poster",
  "content_fingerprint": "sha256:abcd…",
  "state": "pending",
  "attempts": 0,
  "created_at": "2026-06-21T08:00:00+08:00",
  "claimed_at": null,
  "receipt_at": null,
  "not_before": "2026-06-21T12:00:00+08:00",
  "approved": true,
  "payload_ref": "ops/payloads/20260621-fbpost-day011.md",
  "last_error": null
}

狀態機（5 態，單向為主）：

pending → claimed → done
   │         │
   │         └─(claimed 逾 claim_timeout_min 無回執)→ pending（補位，attempts++）
   └─(attempts > max_attempts 或 approved=false)→ dead（停手 + 告警）

1.2 回執 receipt（ops/receipts/<task_id>.json）

由執行的專人自己寫（對賬的另一半），不是 dispatcher 自報——這是 Ohno「異常自停」的眼睛：

{ "task_id":"20260621-fbpost-day011", "result":"posted",
  "posted_fingerprint":"sha256:abcd…", "posted_at":"2026-06-21T12:03:00+08:00",
  "url":"https://...", "by":"w2l-fb-poster" }

1.3 防雙發三層

  1. 檔名即指紋：同 task_id 永遠同檔，重覆派工只覆寫。
  2. 發布前查當日指紋庫 ops/fingerprints/<date>.json：content_fingerprint 已在庫 → 直接 done，不發。
  3. 回執寫 posted_fingerprint：補位前 reconcile 比對「佇列指紋 vs 已發指紋」，相同則判定其實已發、不補。

機制② FREEZE 保守模式

核心思想（Ohno）：機器自己偵測「Jeff 沒在看」、自己收緊、亮一盞燈。Jeff 不需監看；只有「要恢復」時才需他一個動作。

2.1 心跳：怎麼算「Jeff 有在看」

ops/state/attention.json（單一真相）：

{ "last_seen_at": "2026-06-21T09:30:00+08:00", "last_decision_at": "2026-06-20T21:00:00+08:00" }

任何下列事件都呼叫 heartbeat_hook.py 蓋章 last_seen_at：Jeff 開戰情室晨報連結、在 Publer 核可、回覆任一待決卡、跑任一 w2l skill。裁決類事件另蓋 last_decision_at。

2.2 三級狀態機（ops/state/freeze.json + repo 變數 FREEZE_LEVEL）

NORMAL ──(days_since_seen ≥ sok:3)──→ SOFT ──(≥ hard:7)──→ HARD
   ▲                                                          │
   └──────────────(任一心跳事件，立即恢復 NORMAL)──────────────┘
級別 系統行為 告知 Jeff
NORMAL 全功能 無
SOFT（≥3 天） 只發已核可內容；停實驗/A-B/新嘗試；停主動推（待決卡照排佇列不中斷 Jeff） 一次：「你 3 天沒看戰情室，我已切換保守：只發你核可過的、暫停所有實驗與主動打擾。回來看一眼就恢復。」
HARD（≥7 天） 只發已核可且 not_before 到期的剛性排程；停所有 dispatcher 新認領的實驗 lane；完全靜音主動推 一次：「你 7 天沒回，我進入全保守：只維持必發的剛性排程，其餘全部待命，等你回來。」

設計鐵則：FREEZE 只會更保守、不會更激進；恢復只能靠 Jeff 真的出現（心跳），系統不會自我解凍。notify_once 確保同一次收緊只告知一次，不重複煩。

機制③ 省心硬上限

核心思想（Clockwork × Ohno）：保護 Jeff 的安寧是系統的一條剛性產線。超標時砍功能、不砍 Jeff——寧可少推一個提醒，也不破上限。

3.1 定義「主動推」與豁免

  • 算主動推：任何主動把訊息送到 Jeff 眼前、要他注意/決定的動作。分兩型：
  • 待決卡（非中斷）：排進 Jeff 的待辦清單，他有空才看。
  • 中斷型：LINE 即時推播/打斷。
  • 不算主動推（豁免）：① 拉式三報（07/13/21 晨午晚報，是 Jeff 主動去看的脈搏）；② 機制② 的 FREEZE 狀態告知（系統健康告知）；③ 結構性失敗 LINE 告警（保命，永不受上限約束）。

3.2 帳本與預算（ops/state/push_ledger.jsonl，append-only）

每筆主動推前都先過 push_guard.py。每行一筆：

{"ts":"2026-06-21T14:00:00+08:00","date":"2026-06-21","kind":"interrupt","topic":"day011待核可","allowed":true}

當日預算 = 掃 date==今天 的行數：total < max_per_day(2) 且（若 kind=interrupt）interrupt < interrupt_max_per_day(1)。

3.3 唯一出口（強制收口）

所有對 Jeff 的主動推都必須呼叫 push_guard.try_push(kind, topic, payload)——沒有任何程式碼可繞過直接發。守門邏輯：

若 EMERGENCY_STOP/HARD → 一律拒（除豁免類）
若 kind 在豁免清單 → 直接放行，不計帳
否則查當日預算：
  超總上限 → 拒，把訊息降級寫進「明日待決摘要」（不丟失，只是不今天打擾）
  interrupt 超上限 → 自動降級為「待決卡」（不中斷，仍送達）
  通過 → 發送 + append 一筆 ledger

3.4 實作步驟 1–6

  • 步驟 1：寫 scripts/ops/push_guard.py，公開唯一函式 try_push(kind, topic, payload, channel)；回傳 {allowed, downgraded_to, reason}。
  • 步驟 2：實作豁免清單常數（pull_pulse / freeze_notice / structural_alert）——豁免類直接發、不寫預算帳。
  • 步驟 3：實作當日預算讀取（掃 ledger 今日行）＋ EMERGENCY_STOP / FREEZE_LEVEL 短路。
  • 步驟 4：實作降級——interrupt 超標→改待決卡；總量超標→寫入 明日待決摘要（隔天三報帶出，不丟失）。
  • 步驟 5：全域改造：把現有所有「主動發 Jeff」的呼叫點（待核可卡、提醒、各 skill 的中斷）全部改成走 try_push，移除任何直接呼叫通知 API 的舊路徑（單一出口才守得住）。
  • 步驟 6：冒煙測——同日連觸 3 次 interrupt，確認第 1 次發、第 2 次降級待決卡、第 3 次進明日摘要；三報照常不受影響。
